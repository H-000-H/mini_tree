# 全面移植项目版架构回中间件

## Context（背景）

`/home/ning/project/Heterogeneous-Multicore/mini_tree` 是旧版中间件。两个具体项目（stm32f407zgt6、ch32v307）下的 mini_tree 比中间件新约 12.5 小时，是中间件的修改升级版。用户要求**全面按项目版更新中间件**，保持中间件纯洁性。

### 关键事实（已核实）

1. **中间件是旧版**：`hw-instance.h` 中间件仅 10 行，项目版 332 行；中间件 bus 层有 `bus_ops`/`bus_client` 翻译层，项目版已删除改为零翻译直通。
2. **项目版架构是分层的**：
   - **HAL 层**：include 厂商头（`stm32f4xx_hal.h`/`ch32v30x.h`），暴露厂商类型（`SPI_TypeDef*`/`GPIO_TypeDef*`）—— **允许厂商耦合，这是 HAL 的职责**
   - **bus 层**：只 include `hal_spi.h`（HAL 头），**不直接 include 厂商头**，使用 HAL 定义的类型，零翻译直通
   - **vfs 层**：只 include `spi_bus.h`（bus 头），**完全不接触厂商头**
   - **dtsi**：板级硬件描述，允许厂商耦合
3. **两项目架构性文件完全一致**（0 差异）：`board/include/bus.h`、`board/src/bus.c`、`board/include/driver.h`、`board/include/device.h`、`bus/spi/spi_bus.h`——证明这些是平台无关的纯架构文件。
4. **HAL 头设计模式两项目一致**（除厂商头 include 和注释外），都用"嵌入式对象 + DTSI 直投 + 寄存器直写"模式，扁平字段替代旧 `hw_priv_storage` + 二级包装。
5. **中间件 HAL 只有头文件无 .c 实现**（保留 `hal_if_dummy.c` stub）；项目版 HAL 有 `_stm32.c`/`_ch32.c` 实现——这些是项目特定实现，不进中间件。
6. **dtc-lite 头文件解析能力中间件已具备**（`dtc_lite/` 三方一致），仅需更新 `dtc-lite.py` 顶部陈旧注释。
7. **死文件**：`board/CMakeLists.txt`、`bus/CMakeLists.txt` 引用不存在的源文件，顶层不调用。
8. **`bus/include/types/bus_types.h`** 中间件独有但未被任何地方 include（未接线预留抽象，零翻译架构不需要）。
9. **`board/dt-bindings/gpio/gpio-ctl.h`** 中间件独有，项目版没有——项目版 GPIO 已改为 `GPIO_TypeDef*` 直投，旧 GPIO 常量抽象废弃。

### 用户决策

- 中间件是旧版，**全面按项目版移植**（参考 stm 或 ch）
- 只有 hal 层和 dtsi 暴露厂商头是正常的
- bus/vfs/core/board 层不直接 include 厂商头（但可引用 HAL 层定义的厂商类型）
- 注释对齐项目版
- 删除死文件
- `hal/tim/` 不移植（与 `hal/pwm/`、`hal_timer.h` 重叠）

## 移植策略

以 **stm32 项目版为参考**（用户指示"参考 stm 或 ch"，stm32 版更完整），全面覆盖中间件源码。HAL 头用项目版设计（含厂商类型），HAL 实现（`_stm32.c`/`_ch32.c`）不进中间件（由项目提供，中间件保留 `hal_if_dummy.c` stub）。

## 执行顺序

1. **阶段一：删除** — 删除 B 类（死文件/废弃抽象），清理中间件旧版残留
2. **阶段二：覆盖** — 用 stm32 项目版覆盖 A1-A6 文件（核心移植）
3. **阶段三：微调** — A7 注释修正 + `event_bus.cpp` 用 ch32 版 + `compiler_compat.h` 补 ETL 块 + `hal_if_dummy.c` 适配新 HAL 接口
4. **阶段四：验证** — 见下方验证方法

## 风险点

1. **HAL 头 include 厂商头**：项目版 `hal_spi.h` include `stm32f4xx_hal.h`，中间件目录无此头。这是设计意图（HAL 实现由项目提供厂商头路径），中间件 HAL 头不能脱离项目独立编译——与当前设计一致。
2. **`hal_if_dummy.c` 适配**：HAL 头改了接口（`SPI_TypeDef*` 等），dummy stub 可能需更新以匹配新签名。需检查 stub 是否引用了变更的类型。
3. **覆盖时保留中间件已有更先进内容**：个别文件中间件可能已有项目版没有的内容（如 `hal/pwm/hal_pwm.c` 中间件独有），覆盖前需 diff 确认。
4. **`uart_bus.h` 两项目差异 145 行**：stm32 vs ch32 差异大，选 stm32 版作为参考实现，ch32 项目自行适配。
5. **`board_driver.c` 的 `board_safety_add_pin`**：改为 static 后确认无外部引用（已核实无外部调用者）。

## 验证方法

1. **架构纯净性**：`grep -rn "stm32f4xx\|ch32v30x" bus/ vfs/ core/ board/include/ board/src/` → 应为 0（只有 hal/ 和 board/dtsi/ 允许厂商头）
2. **零翻译验证**：`grep -rn "bus_ops\|bus_client_bind\|bus_client_priv" bus/ board/` → 应为 0（翻译层已删）
3. **死文件确认**：`ls board/CMakeLists.txt bus/CMakeLists.txt bus/include/types/` → 不存在
4. **废弃抽象确认**：`grep -rn "bus_types.h\|gpio-ctl.h"` → 应为 0
5. **board_safety_add_pin**：`grep -rn "board_safety_add_pin" board/` → 仅 board_driver.c 内部 static
6. **构建配置**：`cmake -B build` 配置通过
